035 Pass Criteria + 3 Anti-patterns

PR rule · One team member = one dashboard. Every team member builds one dashboard covering their full OKR (all Os, all KRs). Audience = you. Your line manager extracts upward signal by viewing your dashboard — no separate manager view. Below 5 criteria + 3 anti-patterns are read under this rule: management question is a composite driver question; reality check covers all KRs' source coverage; anti-pattern C only triggers when audience spans multiple roles, not multiple Os.

#CriterionDescriptionWhat you write out
01 Management question Must drive a specific decision — not "let me have a look" "My dashboard answers: ______"
02 Action owner Who does what concrete action when numbers move — verb must be executable (not "monitor closely") "When ___ , ___ should ___"
03 Input lineage Where data comes from, who owns it, where it currently lives "From ___ + owner ___"
04 Reality check For 70%+ of your KRs, you can name a concrete data source (system/sheet + owner). "Don't know where data lives" = fail; "exists but messy" = pass (caveats → Gap List) "KR1 data: ___ · owner ___. KR2 data: ___ · owner ___. ..."
05 Diagnostic, not reporting At least one metric answers why, not just what "My why-metric: ___"

3 Anti-patterns · any one triggered = fail

A
Activity counter
Whole dashboard only counts "how much I did", with no "why this happens / how to reduce it".
B
Dataset-first
Starting point is "I have this data", not "I need to answer this question".
C
All-in-one
Serves multiple roles (boss + me + team + external). Single role covering multiple Os is fine — that's the PR rule, not all-in-one.
Pass rule · 5/5 ✓ AND 0/3 anti-pattern → PASS · proceed to P2 HTML draft. Any ✗ → schedule a 15-min 1:1 with your line manager within 1 week (bring filled checklist + P1b output).

What each criterion really means · read this on your own

01 · MANAGEMENT QUESTION

A dashboard exists to drive a decision. Write the question in one sentence. Pass: "Which student onboarding is behind so I know who to call this week?" Fail: "Track team productivity" — that's a topic, not a question.

02 · ACTION OWNER

When a number changes, someone must do something concrete. The verb has to be executable. Pass: "When FCR drops below 60%, ops lead reviews routing rules." Fail: "Monitor closely" / "Stay informed" — these aren't actions.

03 · INPUT LINEAGE

You must name where the data lives + who owns it. Pass: "From Lark base support-tickets, owner Kamohelo." Fail: "From our systems" / "I'll figure it out" — that's not lineage, that's a hope.

04 · REALITY CHECK

The strictest one. For 70%+ of your KRs you can name a concrete data source — system / sheet + owner. Pass: "KR1 from support-tickets base · owner Kamohelo. KR2 from same · owner Kamohelo. KR3 from leave-log · owner HR." Fail: "I don't know where most of this data lives" — that's data-source-survey work, not dashboard work.

05 · DIAGNOSTIC, NOT REPORTING

At least one metric must answer "why", not just "what". Pass: "% of tickets with incomplete student-id" — answers why rework happens. Fail: "tickets resolved" / "calls handled" — these only count what happened.

3 ANTI-PATTERNS · IF YOU TRIGGER ANY, YOU FAIL

A · Activity counter: dashboard only counts what you did, no "why". B · Dataset-first: built around the data you happen to have, not the question. C · All-in-one: serves multiple roles (boss + me + team + external) — pick one role. Single role covering multiple Os is the PR rule, not all-in-one.

Key insight · Components are the last step, derived from the OKR. If you fall in love with a chart type and reverse-engineer the data — that's anti-pattern B (dataset-first).

What each step really means · read this on your own

STEP 1 · OKR

Anchors everything — the strategic outcome the team is trying to move. Without a clear OKR upstream, every downstream step becomes guesswork. Pass: "Reduce repeat-bounce in support tickets". Fail: "Improve support quality" — too vague to derive components from.

STEP 2 · DECISION QUESTION

Translates the OKR into a single specific question the dashboard must answer. The question must drive a decision, not satisfy curiosity. Pass: "Why are tickets bouncing? Which channel + issue type?" Fail: "Show me support stats" — generic, won't shape the dashboard.

STEP 3 · I/O CHAIN

Names who feeds the data in (input owner) and who acts on the output (action owner). When these are different teams, the dashboard becomes a coordination contract — both sides commit. Pass: input = ops fills the ticket; output = Kamohelo adjusts routing rules. Fail: "everyone's data, no one's action".

STEP 4 · SOURCE DATA

Inventory of data sources for each KR — system / sheet + owner. This is your reality check (criterion 4). If you can't name 70%+ of sources, the project is data-source-survey work first, not dashboard work. Pass: KR1 from Lark base support-tickets · owner Kamohelo; channel, issue_type, status, created_at all in.

STEP 5 · HELPER FIELDS

Raw fields rarely answer the management question directly. Helper fields are derived (computed, flagged, joined). Examples: bounce_count, time_to_first_response, resolved_in_one_touch. P3 produces this list — and the missing helpers feed Deliverable B (Gap List).

STEP 6 · COMPONENTS

The visible dashboard — KPIs, charts, tables, filters. This is the LAST step, not the first. Pass: FCR rate top-line + bar by issue_type + drill-down table — each component tied back to step 2's question. Fail: opening Lark and picking pretty chart types — anti-pattern B (dataset-first).

05Group 1 · Top-line + Group 2 · Trend

Anchor question: when you see a chart → which step of the user's visual path does it serve → which Group is it?

①Top-line HealthDefine success per case — then show distance from it

Top-line is not a vague "how is the team doing overall". It's the answer to: for each specific customer type / case / event, what counts as success — and how far am I from it right now? The point is to define success per case, not in aggregate.

Example: a salesperson hits £500k revenue this quarter — looks like success. But if half came from a large-enterprise client placing only small orders (where the actual win is the anchor account placing big orders), that's failure dressed up as success. The dashboard top-line should define success per customer segment and surface that mismatch. Defining success per case is how you reshape thinking and culture.

FAQ · Whole OKR or KR by KR? How does P1a handle a multi-KR OKR?
  • PR rule: one dashboard per team member, covering the whole OKR. P1a evaluates the OKR as a single composite dashboard. Don't split the OKR before P1a.
  • Per-KR fate is decided AFTER P1a passes overall. Some KRs become chart elements; some become progress trackers; some become Gap List rows; some get deferred. All within ONE dashboard. The "is this KR chartable?" question runs at P2/P3, not P1a.
  • Per-KR signals it's NOT chartable: (a) one-off deliverable (e.g., "define 3 timetable models" — once done, nothing to monitor → progress tracker block); (b) no quantifiable signal ("build culture") → narrative section or doc; (c) data doesn't exist and won't be built → defer or do data-prep first; (d) annual / strategic decisions → review doc, not dashboard.
  • For Ano's 8 KRs: ALL go into one Ano dashboard. Likely 5-6 become chart elements; O1 KR3 (define 3 timetable models) becomes a progress tracker block; KRs without nameable sources become Gap List rows feeding KR2 information-quality work. Single dashboard, mixed element types.
FAQ · A KR isn't chartable — what's its place in your single dashboard?

Map your per-KR situation to the signal that triggered "not chartable", then pick a response. The KR still lives in your single dashboard — just as a different element type (progress tracker / gap list row / narrative) or gets deferred.

Signal (a) · One-off deliverable
  • ▸ Reframe as progress tracker — not diagnostic; use Lark Goals / Progress / Milestones. Short-lived, archive when done.
  • ▸ Reframe as ongoing — "define 3 models" → "models adopted / refined ongoing". Risk: force-fit.
  • ▸ Use a doc / status update instead — honest project-management work, not a dashboard.
Signal (b) · No quantifiable signal
  • ▸ Find proxy metrics — team NPS / response time / retention / referral. Warning: proxies get gamed; choose carefully.
  • ▸ Sharpen the question — not "build culture" but "where do we lose people / what's the friction" → naturally converts into a retention / friction dashboard.
  • ▸ Use narrative / qualitative tools — town hall feedback, sentiment analysis. Not a dashboard.
Signal (c) · Data doesn't exist + won't be built
  • ▸ Data prep first — do Deliverable B (input-quality) work, then build the dashboard. This is the KR2 input-quality strategy at heart.
  • ▸ Manual / sample-based — 5 cases per week, hand-filled. Small but real, validates whether the question deserves later automation.
  • ▸ Defer 3 weeks — wait until the data is in place before building.
Signal (d) · Strategic / annual decisions
  • ▸ Track only operational sub-signals in your dashboard — strategic KRs anchor multiple operational sub-signals; chart those in your single dashboard; the strategic decision review lives in a doc, not a separate dashboard.
  • ▸ Doc + dashboard side by side — your single dashboard for daily operational; quarterly / annual review goes to a doc. Two tools, not interchangeable.
