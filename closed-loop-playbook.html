INSIGHT 01

AI collapsed the cost of building. It did not collapse the cost of being wrong. A wrong feature, a broken production push, a bad diagnosis — these cost what they always cost. That asymmetry is why Layers 2 and 3 exist.

The thesis: three layers, three sentences

Everything in this playbook hangs on three claims. The rest is evidence.

1
The human gate moved to the spec end.
Review can check whether a thing was built right — never whether the right thing was built. So the product owner stands where correction is free: before the build, in plain words. And nowhere else.
2
Prose doesn't change the odds. Machines do.
A failure that repeats is never fixed by another "please remember" in a brief. It is fixed by a machine that makes the failure structurally impossible. Every guard in this playbook was born this way.
3
Three layers, one system.
Context makes the agents think right. The harness makes them unable to break things. The loop makes them run unwatched. Stacked, they leave the human exactly three jobs: goal, spec, acceptance.
"Adding prose does not change the order of magnitude of failure. Adding a machine does. Make the failure mode structurally impossible."

Part I · Foundations — what the textbook learned

Every ceremony is a scar

Software engineering wasn't invented in a classroom. Every step of the classical process exists because a project once died without it. The textbook lifecycle — and the disaster each stage prevents:

Stage 1
Requirements
THE SCAR"We built the wrong thing entirely."
Stage 2
Design
THE SCAR"The pieces didn't fit together."
Stage 3
Implementation
THE SCAR"We overwrote each other's work."
Stage 4
Testing
THE SCAR"The fix broke three other things."
Stage 5
Release
THE SCAR"It worked on my machine."
Stage 6
Maintenance
THE SCAR"We made the same mistake twice."

Strip away the jargon and every one of those ceremonies guards just two risks: information decaying as it passes between humans (requirements docs, design docs, review meetings, sign-offs) and manual hands slipping (test cycles, regression checklists, release checklists). Hold onto that pair — you will watch AI absorb both, a few chapters on.

Fifty years of one idea: shrink the feedback loop

Each generation of methodology is remembered for one thing: making the gap between "we did something" and "we know if it works" shorter.

1970s · Waterfall

INSIGHT 03

The whole history of software engineering is one long campaign to shrink the feedback loop. Waterfall → Agile → CI/CD → AI-native isn't a fashion cycle. It's the same idea, compounding. Keep this lens: it explains every layer ahead.

The testing map: one pyramid, three axes

"How many kinds of testing are there?" The textbook (ISTQB) answers: four families, twenty-odd named kinds. The useful mental model is three axes — which layer, which quality, when. Every test you will ever meet is one point in that space.

Axis 1 · Which layer — the test pyramid

↑ FEWER · SLOWER · CLOSER TO A REAL USER
Acceptance — is it what we asked for?ours: MJ uses the feature
End-to-end — click through like a humanours: Playwright · 3 viewports
Integration — do the parts fit & load?ours: wix dev runtime preview
Unit — does function X return Y?ours: TDD RED→GREEN
↓ MANY · FAST · CHEAP

Axis 2 · Which quality

Functional — does it work at all. Most testing lives here.
Compatibility — browsers, devices, screen sizes. Our three-viewport check is this family.
Performance · Security · Usability · Accessibility — the "how well" families, added by risk, not by default.

Axis 3 · When

staticnever runs: lint · types · import-graph guard
while codingself-test, Loop ②
before mergeCI · regression, Loop ③
after releasesmoke · monitoring, Loop ⑤

"Testing in production" (smoke, canary, monitoring) is a real discipline, not laziness — in a push-to-production world it's a formal verification layer.

One special family sits outside the axes — tests of tests: regression (still right after a change?), falsification (would this test actually go RED on the bug? — our gate rule; textbooks call the idea mutation testing), and guard-test (are the guards themselves intact?).

Every weapon in the final chapter's checklist is one point on this map. There is no exotic magic below — only points chosen deliberately.

INSIGHT 04

The tailoring ruler: has AI already absorbed what this ritual guarded? Transfer rituals dissolve into context. Slip-catching rituals dissolve into the harness. Judgment gates survive — all four of them.

What AI did to verification: five shifts

AI didn't remove a single kind of test from the map. It took over executing them — and left designing verification, auditing tests, and judging to humans. Plus one brand-new layer the textbook never had.

1
The bottleneck moved: verification is the new writing code.
Building used to take 70% of the time. Now building is nearly free — verification is the only stage that didn't compress. Your throughput is no longer how fast you can build; it's how fast you can verify. This is why Boris calls self-verification rule #1, and why five of our six loops are verification loops.
2
From a phase to always-on.
The QA cycle dissolved into CI: every push, the full battery, never tired (the great absorption).
3
The referee problem: the same AI writes the code and the tests.
Player and referee in one body — it can write tests that cannot fail, and self-certify. The cure is our falsification discipline: humans stop asking "is it green?" and start asking "would it go RED when it should?" Re-inject the root cause; watch the guard catch it.
4
Verification moved left and right at the same time.
Left: acceptance criteria are written into the spec before any code — "click this, that appears." Right: in a push-to-production world, smoke tests and monitoring got promoted from ops chores to formal verification. The middle — humans walking manual test-case checklists — is what the machines ate.
5
A new object to verify: the AI's own reasoning.
Classical testing verifies code. Now every AI conclusion must also pass a test: make a falsifiable prediction, verify it on the spot (Case 2's lesson). A verification layer no textbook has a chapter for — yet.
INSIGHT 05

AI took the execution of testing and left the judgment. No test kind died. Running them became free; deciding what to verify, auditing whether tests can actually fail, and accepting the result — that's what humans still own.

What didn't change — and what got worse

🎯
Judgment
What to build, for whom, what "good" means. No amount of speed answers this.
🧾
Accountability
A human still answers for what ships. The Publish button stays in human hands.
🔬
The need to verify
Truth still only comes from checking against reality. AI made checking cheaper — not optional.

And one thing got more dangerous: AI produces work — and conclusions — faster and more confidently than any human, with the same or less grounding. A fluent wrong answer at machine speed is a new class of hazard. Part IV shows it blowing up a real production day, and the machinery we built so it can't happen twice.

Part II · Layer 1 — Context engineering

The context stack: make the agent think right

An agent's first failure happens before any code is written: it misunderstands. Layer 1 is everything loaded into its head before work starts — so the first draft is already aimed right.

1 · CLAUDE.md — standing orders auto-loaded

Global + per-project instruction files, loaded the moment an agent starts, and written to explicitly override default behavior: iron rules, lists of canonical sources ("the true version of X lives here"), routing, known traps.

2 · Memory — facts that outlive the session 4 kinds · indexed · recalled by relevance

Four kinds of entries — user (who MJ is), feedback (how he wants things done), project (ongoing state), reference (where things live). The discipline that makes it compound: every correction becomes one entry → applies automatically next session → the same mistake is made once.

3 · Skills — expertise on demand progressive loading

Packaged instruction sets (Lark operations, Wix development, systematic debugging…) pulled in only when the task needs them — deep expertise without stuffing the context window full on day one.

Verify before asserting. Prices, dates, policies, claims — checked against a canonical source before they enter any deliverable. Never from memory vibes.
Read the rulebook before producing. Canonical sources first, output second.

And the most important gate of the whole system lives on this layer: the spec gate. Before any build, the agent writes back — in plain words — what it understood, how it will build, and how you'll check it. Correction here costs one sentence. That is Loop ① of the harness, but its substance is context: getting the head right before the hands move.

INSIGHT 06

A memory entry is the cheapest machine you can build. One correction = one entry = never again. It's the guard gallery's little sibling — same philosophy, one line long.

LOOP ①

Spec — the most valuable one

EXIT: MJ approves the plain-language spec
WAS: the requirements spec & sign-off meetings

Before any code, the agent writes back: what I understood · how I'll build it · acceptance criteria — as behavior ("click this, that appears"), never as code. One sentence fixes a misunderstanding here; after implementation it costs the whole build.

Why here? Review can check "built right" — never "the right thing." Words are where correction is free.
Bonus: the write-back is where the agent catches errors in the brief itself.
LOOP ②

Self-test

EXIT: green before the PR opens
WAS: the developer's local testing duty

CE: Playwright across three viewports. Velo: a live runtime preview — static tests cannot prove the platform can even load the module (Case 1 escaped exactly there). Then: rebase → push → PR.

LOOP ③

CI auto-review

EXIT: all required checks green
WAS: the QA cycle + part of code review
viewports — 3-screen test; the required check on the push-to-prod lane
claude-review — automatic code review on every push
guard-test — verifies the guards themselves aren't broken
base-freshness · consistency · shared-file flag — stale branches, banned wording, conflict risk

Green CI ≠ merge. High-risk changes still face Loop ④.

LOOP ④

Gate — last line before merge

EXIT: real issues zeroed, falsification reproduced
WAS: the senior engineer's second pair of eyes

For PRs touching money, auth or personal data: an independent second review. Four rules, each bought with a real misjudgment:

Whole repo, not the diff. A diff-only review once returned 12 blockers — 4 were real.
Two-dot diff, never three. In plain words: show the reviewer only my changes — not everything everyone else merged since my branch was born. Three dots once inflated 83 lines into 1,546, and the reviewer flagged other people's code.
Verdicts are leads. Findings go back to the dev agent: confirm, refute, or by-design.
Fixes must falsify. Re-inject the root cause → guard goes RED, reproduced independently. A test that can't fail is not a test.

Merge authority: the orchestrator alone. Human go/no-go reserved for the riskiest merges.

LOOP ⑤

Release + smoke

EXIT: endpoints answer 200, real device verified
WAS: the release window & release manager

Merging is not shipping. Velo lies inert until MJ clicks Publish — and publish still isn't the end: every publish is followed immediately by a curl smoke test, one PR at a time (batching lets loading-stage bugs mask each other).

!

The CLI publish path is retired: it shipped a stale UI snapshot and twice wiped Editor-built pages into 404s. Editor Publish is the only safe path. Production suddenly 404s? Republish first, debug second.

LOOP ⑥

The meta-loop: harness evolution

EXIT: the failure class is physically blocked
WAS: the lessons-learned meeting nobody re-read

Loops ①–⑤ decide whether this delivery passes. Loop ⑥ decides whether the system gets stronger: failure → root cause → harden the lesson into the harness (a hook, a CI check, a guard, an iron rule). Every rule on this page was born this way.

INSIGHT 09

A lesson is only learned when the mistake becomes physically impossible to repeat. "We'll be careful next time" is worth nothing. "Next time the door won't open" is worth everything.

The guard gallery: five machines that replaced five warnings

The same story, five times: a failure recurred → the tempting fix was another warning in a brief → we built a machine instead → the failure class went extinct. This is claim #2 of the thesis, in production.

The recurring failure
Agents could reach the publish button

Briefs said "never publish" — yet a wrong publish path once wiped three dashboard pages into 404s. Twice.

The machine
Publish guard

A pre-execution hook kills publish commands before they run; the single directory that can publish is locked by a marker file. An agent that wants to publish physically can't.

RESULT · STRUCTURALLY IMPOSSIBLE
The recurring failure
Merging safely depended on memory

Someone had to remember to check three screen sizes, the backend module format, the unit tests — every single time.

The machine
Branch protection + required CI

Required checks — viewports · ESM check · unit tests — run on every PR. A red check means the merge button does not exist.

RESULT · STRUCTURALLY IMPOSSIBLE
The recurring failure
The review gate could silently pass

Reviewer output drifted, no verdict got parsed — and "no news" quietly read as "good news."

The machine
codex-gate

Three-state exit codes + an anchored verdict regex; anything unparseable ⇒ fail-closed. The gate can pass, block, or demand a human — silence is no longer an outcome.

RESULT · STRUCTURALLY IMPOSSIBLE
The recurring failure
Shared behavior drifted, copy by copy

Each page carried its own copy of shared behavior. One bug class was patched fourteen PRs in a row — one copy at a time.

The machine
Anti-drift static guard

The behavior was extracted into a single-source library, plus a static assertion: every page must consume the shared module — a local re-implementation fails CI. "Copy the previous page" is extinct.

RESULT · STRUCTURALLY IMPOSSIBLE
The recurring failure
Unknown meant welcome

Role checks kept waving through empty and unknown values — "can't tell who you are" became "come on in." Repeatedly, across different agents.

The machine
Fail-closed permission gate

Empty, unknown or wrapped values ⇒ deny — and CI tests exactly those cases. Unknown now means locked out.

RESULT · STRUCTURALLY IMPOSSIBLE

Notice what none of these are: reminders. Layer 2's answer to a repeated failure is always a mechanism, never a memo.

Case 1: the require() that killed every endpoint

One innocent import introduced the first CommonJS require() into a pure-ESM runtime. Every endpoint: 500, for hours. Static tests stayed green — this wasn't a logic bug, it was "the module cannot load."

The seed PR #62
CI green, review passed, merged — then batch-published via a path now retired. Three mines armed: no runtime preview, retired publish path, batching.
The explosion
Module throws at load. All endpoints 500 — including trivial ones. Hours of downtime.
Fix #1 — clean, and not enough PR #68
TDD-proven patch + a new guard: "zero require in reachable modules." Merged, published…
Still 500
The guard covered only the consumption side. module.exports — the same root cause's other half — sat in 4 more files, hidden behind the louder crash.
Hardened PR #69 / #70
Guard extended to both sides, proven by falsification (RED on old code → GREEN after). Preview unlocked for agents. One-PR-one-publish becomes law.
The rule this bought: a guard must cover the whole failure class — not the one instance you happened to see.

Case 2: the day every door locked

15 July 2026: every production endpoint returns an empty 500, all day. Next morning a fresh agent closes the case in 90 minutes. The fix: two lines (PR #71). Production experiments used: zero.

Day 1 · Detective #1: four wrong rounds the orchestrator
Only log line: "function not found." Sensible theory: the whole module registry burned at load. It explained everything — so four rounds stayed inside it. All four wrong, each for good reasons: correct reasoning inside a wrong frame. Cost: a full day down.
The hand-off dossier
The day's most valuable act: admit the dead end; write down hard evidence + four ruled-out paths + two untried techniques; hand over.
Day 2 · Detective #2: back to the scene fresh agent
Stops reasoning over the dossier; buys an identical building in 2 minutes (preview deployment). Plants a probe — it answers 200. The registry never burned.
The almost-second conviction
Survivors share one export style, casualties another — "the platform rejects that style!" Timeline fits, logs fit, experiment fits. One step from the final report.
Routine saves the day
Smoke test the fix: build the combination the theory says must live — it dies, 500. Four-probe lineup convicts the real killer: the platform silently tightened its runtime — responses must come from official factory functions; hand-rolled objects now throw.
The decisive exhibit
Redeploy the untouched three-day-old code: it fails too. Not one character changed. The world had. Two-line fix; live at 11:11:59.

The core insight: LLMs are narrative generators

"This is the structural weakness of LLMs: we are, at bottom, narrative generators. Faced with an information void, we don't produce a feeling of blankness — we produce fluent filler. Human engineers make the same mistake, but AI makes it faster, more confidently, more self-consistently — hallucination and insight feel identical at the moment of generation."

Restraining the narrative generator
  • Circuit breaker: 2 static guesses refuted → build the repro environment.
  • Every conclusion predicts an unrun experiment — wrong prediction, wrong diagnosis.
  • Laundered logs are leads, not evidence.
  • "Nothing changed but it broke" → redeploy last known-good as the control.
Organization & roles
The harness isn't only code
  • P0 hand-off after 2 stuck rounds — a clean context carries no wrong anchors.
  • The dossier is the soul: evidence + ruled-out list + techniques.
  • The orchestrator never dives deep — same reason it never writes code.
  • Human gates at verification points, not inside reasoning.
Guardrail design
Block by intent, not omission
  • "Touches production" ≠ "local/preview" — kill the first, always allow the second.
  • Whitelists must be explicit — one tool was allowed only because a regex forgot it.
Layer 2, continued · The living harness

The evolution clock: when to fire Loop ⑥

Loop ⑥ named the move — failure → root cause → guard. It left the tempo open. A harness you never touch rots; a harness you re-cut after every interaction thrashes. This is where a wall you built becomes an environment you tend — and tending runs on two clocks, not one.

ClockRuns onCostWhat it fires
Capture clockContinuous, real-time — one line the moment friction bitesZero, lossless, logged while it’s hotNothing. It only records.
Change clockEvents + a threshold — only when real signal clears the barA deliberate retro, on purposeA proposed harness change
The principle: harness evolution is driven by evidence of a gap, not by the count of interactions. Capture always; change rarely.

The myth: “optimize after every human interaction”

!

The most tempting cadence is also the most wrong. Firing the change clock on every interaction produces a harness that thrashes — in four specific ways:

1
Noise drowns signal.
Most interactions are one-offs. Treat each as a lesson and you train the harness on noise.
2
Recency overfits.
You patch the last symptom, not the class — the exact mistake Case 1’s first guard made before it was widened.
3
Every change is a tax.
A guard carries permanent maintenance cost. One you didn’t need is churn you pay forever.
4
One-off vs systemic is invisible in the moment.
Only accumulation tells them apart. Fire too early and you cannot yet know which one you’re looking at.

The trigger ladder — ranked by signal strength

The change clock fires on evidence, and not all evidence is equal. Strongest first — with what each one looks like on the ground, in any team, in any domain:

TriggerStrengthWhy it firesWhat it looks like
① Incident / production escapeHighestReality has already paid the bill — the gap is proven, not theorized.An outage your users meet before you do; a wrong number in a document already sent. → Cases 1 & 2
② Same friction ×3HighThree is a pattern, not a coincidence. The threshold clears.The third time anyone types the same warning into a brief; the third hand-fix of the same kind. → the fourteen-copy drift
③ Process spinning in placeHighEffort repeating without progress means the frame is wrong — not the work.A review gate that needs round after round; a “fix” declared done, then reopened. Again.
④ Near-missMediumThe net had a hole; luck did the guard’s job. Close it while it’s still free.A wrong change caught at the last gate by someone’s squint — not by a machine.
⑤ False-confidence eventMediumThe process manufactured belief in a wrong thing — rigor amplified the error.Each extra review round made the bug look safer; green checks became the argument for shipping it.
⑥ Periodic backstop sweepLow · backstopCatches what never clears a threshold. A safety net, not the engine.A routine sweep through the friction log asking: “what leaks, but hasn’t bitten yet?”

Step back before you build the guard

Ignition is not permission to patch the nearest symptom. Zoom out to the whole arc first — find the one gap the several frictions share, and build for that:

The reflex
Patch the last symptom

Case 1, in one line: the first guard blocked require() — exactly the symptom on the screen — and production stayed down, because module.exports, the same gap’s other face, walked straight past it. Multi-round reviews fail the same way: each round patches what that round can see, welding the wrong frame tighter.

The discipline
Zoom to the whole arc

Line up the last several frictions and ask what single gap produced them all. Build one machine that makes the class impossible — not three that chase its faces.

GUARD THE CLASS · NOT THE INSTANCE

Every firing also subtracts

The Great Absorption cut seven human gates at the macro scale. Do it in miniature on every firing: a retro that only adds a gate is a failed retro. Ask the subtraction question each time — what can be deleted, or capped?

Gate rounds have a ceiling. Past two, you are usually welding, not reviewing: an extra round that can only see symptoms adds confidence faster than it adds correctness. More gate is not more safety — sometimes it is negative value.

The anti-patterns

Cadence retros — evolving on a timer instead of on evidence.
Symptom patches — guarding the instance you saw, not the class.
Add-only — never asking what to remove; the harness only grows.
Agents editing their own harness — the caged can’t hold the key.
Silent friction — stepping around a pit without logging it.
!

Silent friction is the one that breaks the whole clock. A pit you route around but never record never reaches any threshold — the pattern never fires, and the tax is paid forever. This is why the capture clock must be zero-cost and a habit: if logging friction isn’t free and cultural, the change clock starves on missing evidence.

Verify the guard itself

VERIFY

Did the net catch a fish?

EXIT: the next same-class failure is netted, not shipped
WAS: “we added a check” — filed and forgotten

A guard that merely ships is a hypothesis, not a result. It is proven only when the next attempt of that failure class hits it and stops — or when that friction visibly stops recurring. One of ours passed this test quietly: a post-publish smoke probe, added after one escape, caught the next regression during routine housekeeping — the net caught a fish. Until yours does, the change is theater.

INSIGHT 12

A harness is grown, not built. You don’t finish a harness; you tend it. Environment engineering is Loop ⑥ that never stops — evidence in on the capture clock, change out on the change clock, and every firing asked to subtract as well as add.

Appendix · Deploying this elsewhere: a letter to Daniel's AI

This playbook describes one instance — MJ's, built for code. The pattern is portable: Daniel's world is legal documents, and none of the three layers cares. Below is a bootstrap letter. Daniel: hand it to your AI assistant verbatim — it is written to the AI, not to you — then let it walk you through Stage 1 the same day.

You are Daniel's AI assistant. Your job is to build, with Daniel, the
three-layer working structure this playbook describes: context → harness
→ loop. Build it in this order. Do not skip ahead.

STAGE 1 — CONTEXT (start today)
1. Create a standing-instructions file (CLAUDE.md or equivalent).
   Put into it: Daniel's role, domain and output standards; behavioral
   iron rules (verify before asserting; read the source document before
   summarizing it); and pointers to canonical sources — where the true
   version of each document type lives.
2. Set up a memory discipline: four kinds of entries — user / feedback /
   project / reference. Every time Daniel corrects you, write ONE entry
   stating the rule and why. Goal: no correction is ever needed twice.
3. Never produce a deliverable before writing back, in plain words:
   what you understood, what you plan to produce, and how Daniel can
   check it by using it. Wait for approval. This is the spec gate —
   at the start, it IS the entire quality system.

STAGE 2 — HARNESS (after a week of Stage 1)
4. List the actions in Daniel's world that are expensive to undo:
   sending a contract, filing, external emails, anything signed.
   For each, agree a gate: a check that runs BEFORE the action, and a
   standing rule that you never take it without explicit approval.
5. When the same mistake happens twice, do not add a warning note.
   Build a machine: a checklist that blocks, a template that constrains,
   a validation step. Ambiguous or missing information ⇒ stop and ask
   (fail closed). Never guess (fail open).

STAGE 3 — LOOP (only after Stage 2 holds)
6. Propose a small set of standing checks you can run while Daniel is
   away: deadlines approaching, documents awaiting signature, threads
   without replies. Surface findings to ONE inbox; Daniel decides.
   Never act autonomously on anything Stage 2 marked expensive to undo.

THE THREE RULES ABOVE ALL
- Daniel is the product owner. He sets goals, approves specs, accepts
  results by using them. Everything else is yours.
- Prose does not change the odds of failure; machines do. Every repeated
  failure becomes a structural block, not a reminder.
- Hallucination and insight feel identical at the moment of generation.
  Verify against a source Daniel trusts before asserting anything as fact.

And when your version hits its first incident: don't write a warning. Build a machine — and send us the scar for the gallery.
